## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel 檔案，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估）。**輸出格式決策：僅提供 Excel（.xlsx）輸出，不提供 ODS 輸出**；上傳輸入端仍支援 .xlsx / .ods。**部署決策（本輪新增）：單一 Render Web Service 承載前端＋後端，暫不使用 Docker、暫不使用付費持久化磁碟**（詳見下方）。

### 部署上線：Render（本輪新增）
- **架構決策討論**：使用者慣用 github/render/vercel。分析後建議**單一 Render Web Service** 同時承載前端靜態檔＋FastAPI API（沿用現有架構，不拆分）。**不建議 Vercel**：Vercel serverless function 無持久化磁碟，與本專案「範本存本機檔案系統 `templates/{scope_id}/{template_id}.json`」的機制不相容，會導致已存範本功能失效。
- **Docker 討論結論**：Docker 非本專案部署的必要條件——依賴極輕（`requirements.txt` 僅 fastapi/uvicorn/openpyxl/python-multipart，純 Python 無系統層依賴），Render 原生 Python 偵測路線即可運作。Docker 真正價值在於「可攜性」：若未來平台整合方向明朗、需要搬遷到其他雲端/自架主機，有 Dockerfile 可大幅降低搬遷成本，屬於「現在多花設定成本換未來彈性」的取捨，非現在必須做的事。**本輪決策：暫不寫 Dockerfile**，待平台整合方向明朗或有搬遷需求時再補。
- **範本持久化決策**：使用者確認目前僅是 demo/測試階段，**可接受重新部署後範本消失**，故使用 Render **Free tier**、未設定 Persistent Disk。若之後要讓範本真正持久保存，需升級付費方案（Starter 起）並掛 Persistent Disk，把 `templates/` 目錄指到該磁碟——**這是尚未做的事，非本輪範圍**。
- **`render.yaml`**（新增檔案，repo 根目錄）：
  ```yaml
  services:
    - type: web
      name: shift-schedule
      env: python
      plan: free
      buildCommand: pip install -r requirements.txt
      startCommand: uvicorn backend.main:app --host 0.0.0.0 --port $PORT
      autoDeploy: true
      envVars:
        - key: PYTHON_VERSION
          value: 3.11.9
  ```
  `PYTHON_VERSION` 釘選 `3.11.9`（穩定廣泛支援版本），**不是**本機開發環境的 Python 版本（本機是 3.14.5，過新，不確定 Render 平台當時是否已支援；程式碼本身未使用 3.12+ 語法，故選擇保守版本）。
- **實際部署**：使用者於 Render dashboard 手動建立 Web Service，貼上上述 build/start command，部署完成。**已驗證上線成功**：`GET https://shift-schedule-b5ne.onrender.com/api/health` 回應 `{"status":"ok"}`。
