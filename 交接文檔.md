## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel/ODS，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。

### 2026-07-14：架構規格討論（使用 grilling skill 逐題確認，Q1~Q13 全部完成）

**流程與UI架構**：
- 單頁式佈局：左側功能欄（店家識別碼輸入、範本選擇、檔案上傳、日期選單、列印、下載）＋右側單日班表主畫面
- 範本重用：上傳前可選現有範本 → 直接套用欄位對照解析；未選範本 → 啟發式猜測 → 彈出確認Modal（欄位對照修正＋次要特徵圈選＋輸出欄位/分組設定）→ 可另存為新範本
- 範本儲存：`templates/{scope_id}/{template_id}.json`，scope_id 為不透明字串，本模組不做身分驗證。**測試階段**：scope_id 由左側功能欄手動輸入欄位提供（已知風險：無碰撞防護，目前單人測試、風險可控）。**正式接上平台後**：改由外部平台注入，資料結構不變
- 範本與檔案格式（ods/xlsx）無關，同一份範本共用兩種格式，後端依實際上傳格式選對應解析器
- 列印＝標準瀏覽器 `window.print()`，非直接印表機整合
- 單日畫面圖表＝瀏覽器端一般JS圖表庫渲染（非原生Excel物件）；「圖表必須原生」規則只限定下載輸出檔案
- 日期選單依上傳檔案實際解析出的天數動態產生
- 下載：可選 ods 或 xlsx（與上傳格式無關），輸出整月檔案，每日一分頁、每分頁一張原生橫向長條圖
- **狀態管理**：後端無狀態（stateless），一次性完整解析整月資料（含異常清單）回傳前端；前端持有整份資料，切換日期/列印皆前端處理，不再打後端；只有「上傳解析」「下載組檔」兩個動作呼叫後端
- **次要特徵圈選UI**：確認Modal內用表格預覽＋每列一個checkbox勾選，非點擊選取或手動輸入列號
- **異常呈現**：解析階段的異常（如區塊列數超出範本預期）採**非阻斷式警告橫幅**，顯示在單日畫面上方，不擋班表顯示；必要欄位在確認Modal階段猜不出來則下拉留空、強制使用者手動選完才能按確認

**資料模型與辨識邏輯**：
- **欄位識別範圍縮減**：只需姓名＋日期＋上下班時間為必要辨識欄位；員工編號、職稱降為「可選辨識、可選顯示」
  - 職稱：可用於分類分組（見下方「外場/內場」修正）
  - 員工編號：部分公司可反推入職日期算年資，但**年資計算本身明確排除在本次範圍外**，只做欄位辨識＋可選顯示
- **重大修正（推翻早期「外場/內場切換鈕」設計）**：工具不認識「外場/內場」這個概念，只認識「一份員工列表＋姓名/日期/時間欄位＋可選分類欄位」。不同使用者分別上傳外場、內場檔案＝兩次互不相干的獨立上傳，工具不需特殊處理。若同一張表已用欄位（如職稱）標記分類，代表使用者要的是**合併在同一張圖**、依分類分區塊顯示，不是切換兩個畫面。是否分組由使用者在確認Modal自選，不自動強制
- **輸出顯示欄位（姓名/時間/員工編號/職稱/分組與否）綁進範本**，在確認Modal一次設定完成，**不做成每次可調整的顯示偏好**——理由：班表有存檔備查需求，同品牌輸出結構須固定
- **員工區塊邊界＝動態偵測**：姓名欄有值→新區塊開始，收集列直到下一個姓名欄有值的列為止（取代原型寫死跳3列的邏輯）
- **區塊內列語意＝範本定義的「預期列數＋固定順序」**：允許實際列數少於預期（視為缺班，非錯誤）；多於預期時不可靜默套用順序解析，標記為異常（非阻斷式警告）
- 員工區塊結構：假設整份表為單一連續區塊，不考慮部門標題列穿插的情境
- **班別數量不寫死為2（早/晚）**：因台灣勞基法規定（連續工作4小時須休息30分鐘），餐飲業實務常見3段班（原3+5小時切割違法後，5小時段須再切一次）。班別數改為**範本可設定參數**，區塊列語意與Gantt圖輔助欄位（base + N組duration/gap）皆用「跑N段迴圈」實作，不設人為上限

### Agent 提出的4項改進建議（2026-07-14，尚待使用者採納/排入範圍）
1. **範本只能新建、沒有編輯既有範本的流程**——公司調整表格版面時，目前只能重新走一次辨識、另存新範本，會導致範本越積越多份雷同的東西
2. **套用既有範本後沒有健檢機制**——如果表格版面悄悄變了但沒觸發「列數異常」，會直接輸出錯誤資料而不自知。建議套用範本後加一層基本合理性檢查（時間欄位能否解析成時間、姓名欄是否為文字等）
3. **次要特徵圈選Modal的預覽範圍未定義**——次要特徵理論上應集中在員工資料上方一小塊區域，若能確認這點，checkbox預覽只需顯示該區域，不用讓使用者在整張表裡找
4. **上傳檔案的安全邊界未討論**——xlsx/ods可內嵌巨集/外部連結，解析時應明確只當純資料讀取，不執行巨集、不追蹤外部參照

## 關鍵設定與上下文 (Key Context & Rules)

- 既有手動版 Excel/ODS 轉換邏輯與 `prototype/convert_day.py` 的欄位假設（供辨識邏輯設計參考基準，非最終規則）：

  - 每人原本假設固定3列（早班起訖/晚班起訖/員工編號），**已確認須改為動態區塊偵測＋範本定義列語意**
- **檔案狀態（使用者確認為手動處理，agent不介入清理）**：`rule.txt`、`班表-7月.xlsx` 已從工作目錄刪除；範例ODS/xlsx測試檔案由使用者自行整理命名，不需agent過問
- 圖表必須是 Excel/ODF 原生 chart 物件（僅限**下載輸出檔案**，網頁預覽不受此限）
- **ODS格式技術踩坑筆記（供未來若需要用程式操作 ODS 檔案參考，2026-07-05驗證）**：
  - `chart:categories` 必須巢狀在 `<chart:axis chart:dimension="x">` 元素內部，直接用 `table:cell-range-address` 屬性即可，不需要 `<chart:domain>` 子元素包裝（已用 OASIS ODF 1.3 官方 RELAX NG schema 驗證）
  - ODF 圖表另外內建一份「local-table」快取快照，是實際渲染畫面的依據；修改儲存格公式或强制重新計算都不會同步更新這份快取，需要直接改寫其 XML 內容
  - Python 沒有成熟的 ODS 寫入函式庫，需直接讀寫 zip 內的 `content.xml` / `META-INF/manifest.xml` / `Object N/`；`mimetype` 檔案必須是 zip 內第一個項目且不可壓縮（ZIP_STORED）
  - Windows 上對已被 LibreOffice/OpenOffice 開啟的 ods 檔案寫入會遇到檔案鎖定，建議寫暫存檔、等使用者確認關閉檔案後再取代正式檔名
- 泛用性辨識機制總原則：主要特徵欄位位置不固定，機制為「啟發式猜測→使用者確認畫面（下拉選單修正）」，不追求全自動辨識；欄位對照確認後存成範本，範本歸屬由使用者手動選擇（不做自動指紋比對配對）
