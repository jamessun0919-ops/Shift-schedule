## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel 檔案，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估）。**輸出格式決策：僅提供 Excel（.xlsx）輸出，不提供 ODS 輸出**；上傳輸入端仍支援 .xlsx / .ods。

### 甘特圖新增「每小時上班人數」顯示（本輪，使用者需求，已完成且驗證通過）

- **需求**：xlsx 甘特圖新增每小時上班人數，顯示於時間軸上方。
- **統計規則**：資料驅動（沿用 `employees[i]["days"][d]["shifts"]` 已解析出的班別起訖時間），對每個整點小時 h 做**區間重疊**判斷——`shift.start < h+1 且 shift.end > h` 即算該員工這小時上班，非整點瞬時判斷。使用者確認過此規則，也確認「資料驅動」優於「視覺計算圖表橫線數量」（後者等於把前者已算出的答案繞更間接、更依賴渲染引擎的路再算一次，無額外好處）。
- **顯示機制**：姓名軸最上方新增一列虛擬類別「上班人數」（工作表第 2 列，真實員工資料從第 3 列開始），沿用甘特圖既有座標系統（每小時一個固定 1 小時寬度的區段），保證與時間軸整點刻度對齊；區段本身透明無色，只用自訂資料標籤（`DataLabel`）顯示人數文字。此方案（做法A）優於「脫離繪圖區、飄在軸線外側」（做法B，需手動換算 EMU 座標且無法保證跨軟體對齊，與先前「軸線位置」踩坑同一類風險），已與使用者確認採用做法A。
- **實作位置**：`backend/generator.py`
  - `compute_hourly_headcount(employees, day, axis_start, axis_end)`：純資料運算，回傳每小時人數列表。
  - `generate_xlsx()`：姓名軸第2列寫入「上班人數」標籤，真實員工列位移（`r = idx + 3`）；新增每小時輔助欄（`cnt{h}`），真實員工列全填 0，「上班人數」列依序填 1/24（固定 1 小時長度）。
  - `add_xlsx_gantt_chart()`：新增每小時人數 series（透明、`DataLabel(idx=0, numFmt=f'"{count}"', showVal=True, ...)`），`chart.height`/`last_row` 相關計算同步加 1（多一個虛擬類別）；`chart_col` 欄位英文字母換算改用 `_col_letter()`（原本 `chr(65+n)` 在欄位數超過 26 時會產生非法欄位代號，本次新增欄位後必然超過）。
  - `_fix_datalabel_numfmt_sourcelinked(xlsx_path)`：**存檔後 XML 後處理函式**，見下方關鍵設定說明。
- **驗證**：小規模對照組（Alice/Bob 兩人、8:00-12:00 四小時軸）與真實範例資料（5人／10人／30人）逐一 LibreOffice headless 渲染確認人數與長條對齊正確、無疊字亂碼；`tests/verify_engine.py` 6 份範例全數 `healthy=True` 無回歸；直接打 `/api/convert` API 解壓確認正式流程輸出正確套用所有修復。**本輪未經使用者本人瀏覽器實測**（僅 agent 端 headless 渲染 + API 驗證），下次應請使用者實際操作確認視覺效果。

## 關鍵設定與上下文 (Key Context & Rules)

- **技術定位**：欄位辨識全部為規則式手寫邏輯，不是機器學習模型。完整規則表見 `README.md`。
- **架構決策：FastAPI + 獨立微服務**（2026-07-15 確認）。**輸出格式決策：僅 Excel，不提供 ODS**（2026-07-21 確認，上傳輸入不受影響）。
- **範本 JSON schema 的 `display` 區塊**：`{"axis_start": int, "axis_end": int}`（整點小時，0~24）。讀取點：`frontend/app.js` 的 `renderDayGantt()`、`backend/generator.py` 的 `generate_xlsx()`、`backend/heuristics.py` 的 `guess_axis_range()`；找不到時一律 fallback 8/24。
- **月份不是範本欄位**：`month` 是 `parse_schedule()` 每次呼叫時當場算出來的回傳值，**不會**寫進範本 JSON、也不會透過 `/api/templates` 存起來。frontend 對應用一個獨立的模組層狀態 `currentMonth`，在每次 `/api/preview` 回呼時重新賦值。**未來若有類似「這次分析出的資訊」需求，優先考慮走這個「回應層欄位」模式，而非塞進範本結構。**（本輪「每小時人數」延續同一模式：`compute_hourly_headcount()` 也是每次 `generate_xlsx()` 呼叫時當場算出，不快取、不寫入範本。）
- **甘特圖姓名軸新增「上班人數」虛擬列（本輪新增，重要）**：工作表第 2 列固定保留給這個虛擬類別，真實員工資料從第 3 列開始（`r = idx + 3`，非原本的 `idx + 2`）。之後若要再調整這個區塊的程式碼，務必注意所有「列位置」相關計算（`last_row`、`chart.height` 的類別數、`cats` Reference 範圍）都已經把這一列算進去，不要漏掉。
- **openpyxl BarChart 的軸對應關係（務必記住）**：`chart.x_axis` 不論 `chart.type` 是 `"bar"`（橫向）或 `"col"`（縱向），永遠序列化成 `<c:catAx>`（類別軸，本專案對應姓名）；`chart.y_axis` 永遠是 `<c:valAx>`（數值軸，本專案對應時間）。這跟畫面上長條實際橫向或縱向排列**無關**。
- **openpyxl `chart.set_categories()` 呼叫時機**：必須在所有 `chart.add_data()` 呼叫**之後**才呼叫，否則分類資料不會出現在輸出檔案裡。
- **openpyxl 隱藏欄位＋圖表連動**：`ws.column_dimensions[letter].hidden = True` 隱藏欄位後，若該欄位是圖表資料來源，**必須**同時設定 `chart.visible_cells_only = False`，否則圖表預設只畫可見儲存格、資料會憑空消失。
- **openpyxl 圖表高度**：`add_xlsx_gantt_chart()` 內 `chart.height = max(10, n_employees + 1) * 0.8`（本輪把 `+1` 算進去，因為姓名軸多了「上班人數」這個類別，別再改回 `n_employees` 不加 1，否則圖表會偏矮、姓名標籤可能被跳過）。**這行必須是函式內最後一次對 `chart.height` 賦值**，先前曾發生過重複賦值覆蓋掉修復的 bug（已排除），未來若要調整圖表尺寸，先搜尋整個函式確認沒有重複賦值。
- **欄位英文字母換算務必用 `_col_letter()`，不要用 `chr(65+n)`**：本輪新增每小時人數輔助欄後，輔助欄總數（`base` + `dur/gap` + 每小時 `cnt`）很容易超過 26 欄，`chr(65+n)` 換算超過 26 會產生非法欄位代號（`add_xlsx_gantt_chart()` 內的 `chart_col` 已改用 `_col_letter()`，之後任何新增欄位的程式碼都要延續這個習慣）。
- **【本輪重要教訓】openpyxl 的 `DataLabel.numFmt` 只能設定 `formatCode`，沒有介面可設定 OOXML 規格要求的 `sourceLinked` 屬性**：若不明確標示 `sourceLinked="0"`，Excel／LibreOffice 會判定「沿用來源儲存格格式」而**忽略自訂格式碼字面值**（用來把 1 小時長度的區段值顯示成人數數字的技巧，見 `add_xlsx_gantt_chart()` 的每小時人數 series）。解法：`generate_xlsx()` 在 `wb.save()` **之後**呼叫 `_fix_datalabel_numfmt_sourcelinked(output_path)`，重新打開存好的 xlsx（zip 檔），用正規表示式對圖表 XML 裡我們自己寫入的自訂格式碼欄位（`formatCode` 值以 `&quot;` 開頭，即引號字面值）補上 `sourceLinked="0"`，再寫回檔案。**這是這個專案第一次採用「存檔後修改 XML」的手法**（先前都只有讀取驗證，沒有寫入修改過），若之後还有類似「openpyxl API 沒有暴露某個 OOXML 屬性」的需求，可以參考這個模式。
- **【本輪重要教訓】自訂 `DataLabel` 務必明確關閉 `showCatName`/`showSerName`（連同 `dLbls` 這一層一起關）**：不明確關閉，LibreOffice 會把類別名稱／數列名稱一起串進標籤文字（如「上班人數;cnt8;0.0417」），且這個關閉不能只設在有自訂內容的那個點（`DataLabel(idx=0, ...)`），要在 `DataLabelList`（`dLbls`）這一層也設，否則「未被 idx=0 覆蓋的其他點」（即真實員工列的人數欄=0 的點）會落回「未設定=顯示」的預設值，在真實員工的長條上疊出多餘文字。完整範例見 `add_xlsx_gantt_chart()` 內 `dlbls_common` 那段程式碼。
- **openpyxl 類別軸標籤間隔**：`chart.x_axis.tickLblSkip`／`tickMarkSkip` 設為 `1` 理論上強制全部顯示，但 LibreOffice 並不會真正遵守這個屬性，仍會依圖表實際高度自行算間隔。此設定予以保留（無害）但不能單獨依賴它，真正解法是拉高 `chart.height`。
- **openpyxl 移除圖例**：`chart.legend = None`（預設是一個 `Legend()` 物件，會顯示在圖表右側）。
- **調整既有數值軸的 `crosses`/`axPos`/`tickLblPos` 屬性來改變軸線顯示位置，在 LibreOffice 與 Google 試算表上都被證實無效**（屬性正確寫入 XML，但渲染結果完全不受影響）。若未來還有「把某條軸換到另一側/另一端」的需求，直接採用次座標軸技術，不要再花時間嘗試調整既有單軸的這幾個屬性。
- **openpyxl 次座標軸（雙軸圖表）技術**：`chart1 += chart2` 可以合併兩個圖表物件的繪圖區。完整範例見 `backend/generator.py` 的 `add_xlsx_gantt_chart()` 內 `top_axis_chart` 那段程式碼與其註解。
