## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel 檔案，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估）。**輸出格式決策：僅提供 Excel（.xlsx）輸出，不提供 ODS 輸出**；上傳輸入端仍支援 .xlsx / .ods。**部署決策：單一 Render Web Service 承載前端＋後端，暫不使用 Docker、暫不使用付費持久化磁碟**。

### 新增功能：外部受測者上傳新樣本自動存檔進 git（本輪核心工作，尚待 Render 環境驗證）
- **需求**：受測者上傳的檔案，若檔名不存在於 repo 的 `data/`，同步存檔進倉庫，供日後累積真實樣本驗證欄位辨識規則。
- **確認過的架構決策**（詳見對話紀錄三輪討論）：
  1. 本機開發環境與 Render 正式環境都要生效。
  2. 要真正寫進 git（不是只存磁碟）——因為 Render 免費方案磁碟非持久化、`autoDeploy: true` 使任何其他推送都會觸發重新部署清空磁碟，純磁碟暫存撐不到人工審查的時間點。
  3. 不要「不經審查直接自動 commit 進正式 `data/`」，改成先存進獨立的 `pending_samples/` 資料夾並自動 commit/push；開發者事後手動把核可的樣本搬進正式 `data/`。
  4. 「已存在」判定用檔名比對（不比對內容雜湊）。
  5. Render 正式環境需要一組僅限此 repo、僅 Contents 讀寫權限的 GitHub Fine-grained Personal Access Token，存於環境變數 `GITHUB_PUSH_TOKEN`；本機開發環境沿用既有 git 憑證，不需要 token。
- **實作內容**（`backend/main.py`）：三個上傳端點（`/api/analyze`／`/api/preview`／`/api/convert`）共用的 `_save_upload_to_temp()` 內新增檔案檢查：
  - `_archive_sample_if_new(filename, contents)`：檔名不在 `DATA_DIR` 也不在 `PENDING_SAMPLES_DIR` 才寫入 `pending_samples/`，並用背景執行緒（`threading.Thread`，不阻塞主回應）呼叫 `_git_commit_and_push_sample()`。
  - `_git_commit_and_push_sample(safe_name)`：`git add`／`git commit`（用固定的 `sample-archiver` 身分，非開發者個人身分）／`git push`；有 `GITHUB_PUSH_TOKEN` 環境變數時組出帶 token 的 remote URL 推送，否則用預設 `origin`（本機情境）。任何步驟失敗只用 `logger.exception()` 記 log，不會讓使用者的上傳/分析/轉換流程失敗。
  - `.gitignore` 新增 `!pending_samples/*.xlsx`／`!pending_samples/*.ods` 例外規則（原本 `*.xlsx`／`*.ods` 全域排除，已用 `git check-ignore`／`git status --porcelain` 驗證例外規則生效）。
- **已驗證**：monkeypatch 掉真正的 git 呼叫，純測 dedup／寫入邏輯三種情境（新檔名／同檔名重複上傳／檔名已在 `data/` 中）皆正確；過程中**未觸發任何真的 git push**（避免在功能未經使用者審視前污染正式 git 歷史）。
- **尚未驗證**：真實 Render 環境下 `git push`（含 token 認證）是否真的成功——包含 Render 的執行環境是否真的存在 `.git` 目錄與可用的 `git` 執行檔（原生 Python 部署，非 Docker，理論上應該有，但未實測）。
- **已知限制（已告知使用者）**：
  1. `data/` 本身依既有規則從未推送進 git（`.gitignore` 排除），所以 Render 上的部署永遠不會有 `data/` 內容，Render 端的「已存在」比對實際上只對 `pending_samples/` 有效，對開發者本機已收錄的樣本無感知（可能造成同名真實樣本在 Render 端被重複收進 `pending_samples/`）。
  2. 本機開發環境測試上傳新檔名樣本，也會一併自動 commit/push（使用者已知悉並選擇兩者都要，非僅 Render 才會觸發）。
