## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel/ODS，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估）。

## 已完成進度 (Completed)

### 核心引擎：`backend/heuristics.py` / `backend/parser.py` / `backend/generator.py`
- 全部欄位辨識為規則式（rule-based）統計判斷，不使用機器學習、不使用寫死的職稱關鍵字清單。技術手法與各欄位判斷規則已整理成表格寫入 `README.md`〈欄位辨識規則技術參考〉章節。
- 日期列、姓名欄位、員工區塊列數、區塊內姓名列偏移、員工編號欄位：皆為自動偵測。N 段班泛化（區塊列語意不寫死班數，最後一列=行政/員編列，其餘依序為班別0~N）。
- `get_hidden_columns()`：偵測來源檔案中被隱藏的欄（xlsx 讀 `column_dimensions`、ods 解析 `table:visibility`），供前端預覽格線跳過渲染用，純顯示層資訊，不影響解析。
- **`guess_template()` 的「首日儲存格」猜測邏輯已通用化並統一到後端**：原本前端用簡化規則（找欄內第一個數字/格式正確的儲存格）在真實案例中會誤判——若第一位員工第一天恰好排休假，內容格式不像正常班別，就會被跳過而猜到錯誤的列。現改為用既有的姓名區塊偵測（`find_block_anchors`，原本只有「3列以上」分支才用）統一套用到所有 `expected_rows`（1/2/N 列皆可，含跨區塊一次跳一整個區塊），直接找「最早一列有效姓名列」對應到「第一個班別型別列」的實際列號。新增回傳欄位 `first_data_row`（1-based，純顯示用途，不影響解析、不寫入範本 schema）。
- `tests/verify_engine.py` 涵蓋 `白霧.ods` 與 5 份不同格式真實範例班表，全數 `healthy=True`。

### 後端：`backend/main.py`（FastAPI，無狀態 REST API）
- 端點：`POST /api/analyze`（回傳新增 `hidden_cols`）、`POST /api/preview`、`POST /api/convert`、範本 CRUD、`GET /api/health`。
- 無狀態設計：`/analyze`、`/preview`、`/convert` 皆需每次重新上傳原始檔案，不留伺服器端暫存/session。
- 邊界防呆：副檔名白名單、空檔案、20MB 大小上限、template JSON 格式錯誤、output_format 白名單、scope_id/template_id 路徑安全字元驗證。
- **xlsx 檔案鎖定 bug 已修復**：`heuristics.py`／`parser.py` 讀取 `.xlsx` 時用 `openpyxl.load_workbook(..., read_only=True)`，先前未呼叫 `wb.close()`，導致 Windows 上暫存檔案控制代碼未釋放，`os.remove(tmp_path)` 噴出 `PermissionError: [WinError 32]`，使 `/api/analyze` 對 `.xlsx` 檔案回傳 500。已改為 `try/finally` 搭配 `wb.close()`。只影響 `.xlsx`（`.ods` 走 `zipfile` context manager 自動關閉），且只有走過瀏覽器實際上傳流程才會觸發，`tests/verify_engine.py` 直接呼叫函式測不到。
- **甘特圖長條統一單色（xlsx 下載檔案）**：`add_xlsx_gantt_chart()` 原本沒有明確設色，靠 Excel 自動依 series 編號配色，導致班別1/2 顏色不同。新增常數 `DUR_SERIES_COLOR = "2A78D6"`（對應網頁預覽 `--series-1`），明確幫每個班別 series 設定 `solidFill`。**ods 下載檔案不需要改**：`generate_ods()` 只重新指定既有圖表 series 的儲存格範圍，從不動 `chart:style-name`；`白霧.ods` 範本內嵌的圖表本來就讓所有班別 duration series 共用同一個樣式（`ch7`，橘紅色 `#ff420e`），已經是統一色，只是色號跟 xlsx/網頁的藍不同（未動範本檔案本身，見下方「已知的可選優化」）。

### 前端：`frontend/index.html` / `style.css` / `app.js`（原生 HTML/CSS/JS，無框架）
- 由 FastAPI 於 `/` 掛載為靜態網站，本機開發不需另開前端 server。
- 上傳表單、確認 Modal（原始資料格線預覽 + 通用區塊列語意編輯器，每一列可自由選 5 種類型）、單日預覽橫向長條圖、`window.print()` 列印、轉換下載。
- **步驟編號**：Modal「確認欄位對照」與「單日預覽 & 下載」的區塊編號，已從照頁面位置排的「1／3／2」改為與操作動線一致的「1／2／3」。
- **確認欄位對照 Modal 文字潤飾**：兩輪下來已全部完成，欄位對照區文字統一為口語化提問（直＝欄、橫＝列）。
- **隱藏欄位不顯示**：`renderPreviewGrid()` 依 `hidden_cols` 跳過隱藏欄的表頭與儲存格渲染，但 `td.dataset.col` 仍保留真實欄號，欄位對照數值不受影響。
- **欄位對照表單易讀性**：`map-name-col`（姓名所在欄）、`map-first-day-col`（上下班起始格）改顯示欄字母/完整儲存格參照（如 `D`、`C5`，列號由後端 `first_data_row` 提供）；`map-cols-per-day` 提問文字重寫；`map-name-row-offset` 改人類直覺的 1-based 輸入（`openModal()`/`collectTemplateFromForm()` 做 +1/-1 轉換，**範本 JSON schema 仍是 0-based**）；員編列（metadata）移除不必要的「欄位名稱」輸入框（固定 `employee_id`），「欄位位置」改欄字母；另存為範本名稱預設值改「品牌文字＋範本」（已存範本則沿用原名）。
- **單日預覽甘特圖顯示所有員工**：原本會把當天沒有有效班次的員工整列隱藏，與下載檔案（本來就列出所有員工）行為不一致；已修正為網頁預覽也列出全部員工，沒有班次的人只留姓名、空長條軌道。

## 關鍵設定與上下文 (Key Context & Rules)

- **技術定位**：欄位辨識全部為規則式手寫邏輯，不是機器學習模型。完整規則表見 `README.md`。
- **架構決策：FastAPI + 獨立微服務**（2026-07-15 確認，已寫入 README）：無狀態單一職責的 I/O bound 服務適合原生 async；不需要 Django 的 ORM/Admin/Session。**「獨立服務」是技術/部署層面決定，不等於「獨立維護」（組織/人力層面）**。
- **`/api/preview` 回傳完整 `employees` 陣列**，供前端單日預覽自由切換任一天。
- **確認 Modal 區塊列語意編輯為「完全通用版」**：每一列可自由選 5 種類型（shift/shift_start/shift_end/shift_string/metadata）。
- **前端無框架**：原生 HTML/CSS/JS，FastAPI 直接掛載靜態檔案，無 build step。
- **員編判定門檻**：命中率 ≥ 50% 的區塊數才採用該欄位，寫死於 `backend/heuristics.py`，尚未依真實資料調整。
- **範本儲存結構**：`templates/{scope_id}/{template_id}.json`，`scope_id` 由使用者上傳頁手動輸入，**目前無歸屬驗證**（見上方部署待辦第 3 項）。
- **姓名長度限制**：判定為姓名候選的字串最長 20 字元。
- **xlsx 讀檔資源釋放**：`backend/heuristics.py`、`backend/parser.py` 的 `openpyxl.load_workbook(read_only=True)` 皆已改為 `try/finally` 搭配 `wb.close()`，未來若新增其他 `load_workbook` 呼叫點，須比照辦理，否則 Windows 上會重現暫存檔刪除失敗的問題。`get_hidden_columns()` 對 xlsx 用**非** read_only 模式另開一次 workbook（read_only 模式不提供 `column_dimensions`），一樣要記得 `wb.close()`。
- **欄位對照表單的欄字母顯示是純前端轉換**：`map-name-col`、`map-first-day-col` 顯示給使用者看的是欄字母/儲存格參照，但送往後端的 `template.mapping.name_col`/`first_day_col` 仍是數字欄號（用 `refToCol()` 轉換）；`map-first-day-col` 附帶的列號來自後端 `first_data_row`，純粹是猜測給人看的參考值，後端解析完全不使用這個列號。`map-name-row-offset` 同理，UI 上 1-based、送出前 JS 轉回 0-based，**範本 JSON schema 未變**。
- **甘特圖顏色**：`frontend/app.js` 的 `SERIES_COLORS` 陣列仍保留 4 色（藍/綠/洋紅/黃），但目前網頁預覽與 xlsx 下載都固定只用第一色（藍），因為前後段班別性質上無差異、不需色彩區分；`backend/generator.py` 的 `DUR_SERIES_COLOR` 常數同步對應。若未來需要真的區分班別顏色，兩處都要改回去。
- **Gantt 圖表原生需求**：下載輸出檔案為 openpyxl/ODF 原生圖表物件；網頁預覽為 CSS 繪製的長條圖，僅供瀏覽/列印使用。
- **ODS 輸出固定以 `白霧.ods` 作為圖表骨架來源**，資料本身來自使用者上傳檔案解析結果；圖表樣式（含顏色）沿用範本既有定義，程式從不覆寫。
- **檔案安全**：解析時只當純資料讀取，不執行巨集、不追蹤外部連結；上傳檔案有 20MB 大小上限與副檔名白名單。
- **Git 遠端倉庫**：`https://github.com/jamessun0919-ops/Shift-schedule`（已設為 origin）。
- **推送規則**：僅推送紀錄檔案（交接文檔/工作日誌/對話紀錄/README）與原始碼（`backend/`、`frontend/`、`prototype/`、`tests/*.py`、`requirements.txt`）；測試/範例用資料檔案與 `__pycache__`、`templates/` 依 `.gitignore` 不推送。
