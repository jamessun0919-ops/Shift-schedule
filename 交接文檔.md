## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel 檔案，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估）。**輸出格式決策（2026-07-21）：僅提供 Excel（.xlsx）輸出，不再提供 ODS 輸出**；上傳輸入端仍支援 .xlsx / .ods。

## 目前的瓶頸或停頓點 (Current Blocker/Status)

**唯一的實質阻塞點：xlsx 甘特圖姓名軸標籤跳過問題，本輪未解決。**

- **問題現象**：使用者上傳真實檔案（`MR08裕隆城甘特圖班表3月.xlsx`，13 位員工，範本 scope_id=`MORE`、`display: {axis_start:10, axis_end:22}`）轉換下載後，甘特圖姓名軸只顯示 7 位員工（每隔一位跳過一位），其餘 6 位姓名標籤消失。**資料本身沒有遺漏**：試算表資料格、圖表底層長條位置都正常，只有姓名軸的文字標籤沒顯示。
- **根因定位**：Excel/LibreOffice 的長條圖類別軸（姓名軸）有「標籤間隔」設定，預設值為「自動」——當類別數（員工數）較多、圖表高度顯得不足時，渲染軟體會自動計算並跳過部分標籤以避免重疊，這不是資料錯誤，純粹是顯示層的自動行為。
- **已套用的修復**：`backend/generator.py` 的 `add_xlsx_gantt_chart()` 新增 `chart.x_axis.tickLblSkip = 1`、`chart.x_axis.tickMarkSkip = 1`，強制間隔為 1（每個類別都顯示），理論上不應再讓渲染軟體自行決定跳過。
- **獨立驗證確認修復有效，但使用者實機測試仍失敗**：
  1. 用 Python 直接呼叫 `generate_xlsx()`（不經過 API/伺服器）測試 30 位員工的合成資料（`範例班表1MORE.ods`）與**使用者實際使用的同一份檔案＋同一份範本設定**（`MR08裕隆城甘特圖班表3月.xlsx` + `templates/MORE/e59e7d978fa8.json`），用 LibreOffice headless 轉 PNG 直接渲染，兩次測試皆確認全部姓名正確顯示、無跳過。
  2. 但使用者透過網頁完整流程（上傳→分析→確認→轉換→下載）實際測試兩次（第一次懷疑是用到 port 8000 的殘留舊版伺服器；第二次已確認伺服器 log 顯示走的是重啟後、含修復的新版本 port 8010），**畫面截圖顯示的姓名軸跳過情況與修復前完全相同（7/13）**。
  3. 本輪未能查出這個「獨立測試成功、實機仍失敗」的落差原因。可能方向（皆未驗證）：
     - 瀏覽器下載的檔案是否為最新版本（瀏覽器下載管理員的重複檔名處理、或使用者開啟的是舊的本地副本而非新下載的檔案）。
     - LibreOffice 互動式開啟 Calc 畫面 vs. headless 批次轉檔的渲染路徑是否有差異（例如互動模式下，標籤間隔的「自動」計算可能是即時依當下視窗寬度/縮放動態運算，理論上 `tickLblSkip=1` 應該會覆蓋掉自動行為，但這一點沒有實際在互動式 Calc 開啟驗證過，此環境只做了 headless 轉檔驗證）。
     - 本輪過程中兩度發生本機測試 server 環境異常（port 8000 曾出現指令列工具查不到、也殺不掉的殘留進程，`Get-Process`/`Get-CimInstance` 都查無此程序但 port bind 測試證實仍被占用；使用者一度表示已移除又重裝本機 Python），不確定是否與此有關，也不確定是否已完全排除。
- **下一步建議**（見下方「下一步行動」）：直接向使用者索取「仍然顯示不全」的實際下載檔案本身進行位元組比對或直接解壓 XML 檢查 `tickLblSkip` 是否真的寫入，而非僅依賴螢幕截圖，才能確認是「修復沒生效」還是「修復生效但渲染端仍跳過」兩種完全不同性質的問題。

## 關鍵設定與上下文 (Key Context & Rules)

- **架構決策：FastAPI + 獨立微服務**（2026-07-15 確認）。**輸出格式決策：僅 Excel，不提供 ODS**（2026-07-21 確認，上傳輸入不受影響）。
- **範本 JSON schema 的 `display` 區塊**：`{"axis_start": int, "axis_end": int}`（整點小時，0~24）。讀取點：`frontend/app.js` 的 `renderDayGantt()`、`backend/generator.py` 的 `generate_xlsx()`、`backend/heuristics.py` 的 `guess_axis_range()`；找不到時一律 fallback 8/24。
- **openpyxl BarChart 的軸對應關係（務必記住）**：`chart.x_axis` 不論 `chart.type` 是 `"bar"`（橫向）或 `"col"`（縱向），永遠序列化成 `<c:catAx>`（類別軸，本專案對應姓名）；`chart.y_axis` 永遠是 `<c:valAx>`（數值軸，本專案對應時間）。這跟畫面上長條實際橫向或縱向排列**無關**。
- **openpyxl `chart.set_categories()` 呼叫時機**：必須在所有 `chart.add_data()` 呼叫**之後**才呼叫，否則分類資料不會出現在輸出檔案裡。
- **openpyxl 隱藏欄位＋圖表連動**：`ws.column_dimensions[letter].hidden = True` 隱藏欄位後，若該欄位是圖表資料來源，**必須**同時設定 `chart.visible_cells_only = False`，否則圖表預設只畫可見儲存格、資料會憑空消失。
- **openpyxl 類別軸標籤間隔**：`chart.x_axis.tickLblSkip`／`tickMarkSkip` 預設 `None`（對應 Excel/LibreOffice 的「自動」，類別數多、圖表高度不足時會自動跳過部分標籤）；設為 `1` 理論上強制全部顯示。**本輪已套用但使用者實機測試未確認生效，見上方「目前的瓶頸」**。
