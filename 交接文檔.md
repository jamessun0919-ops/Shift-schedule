## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel/ODS，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估，詳見下方「關鍵設定與上下文」）。

### 後端：`backend/main.py`（FastAPI，無狀態 REST API）
- 端點：`POST /api/analyze`（上傳分析）、`POST /api/preview`（試解析健康檢查，回傳**完整** `employees` 陣列供前端渲染任一天預覽）、`POST /api/convert`（產生下載檔，xlsx/ods）、範本 CRUD（`GET/POST /api/templates`、`GET/PUT/DELETE /api/templates/{scope_id}/{template_id}`）、`GET /api/health`。
- 無狀態設計：`/analyze`、`/preview`、`/convert` 皆需每次重新上傳原始檔案，不留伺服器端暫存/session。確認 Modal 內欄位微調只在前端狀態，只有按下「確認」才呼叫一次 `/preview`。
- 邊界防呆：副檔名白名單（.xlsx/.ods）、空檔案、20MB 大小上限、template JSON 格式錯誤、output_format 白名單、scope_id/template_id 路徑安全字元驗證（防路徑穿越）。
- 範本儲存：`templates/{scope_id}/{template_id}.json`，`scope_id` 為使用者上傳頁手動輸入的店家/品牌文字（無帳號系統，測試階段用手動輸入代替）。
- 已用真實檔案（範例班表1.ods）跑過完整 end-to-end：analyze→preview→convert（xlsx 31分頁、ods zip結構皆驗證）→範本 CRUD 全生命週期（含中文 scope_id）→6 種錯誤邊界情境，全數正確。

## 關鍵設定與上下文 (Key Context & Rules)

- **技術定位**：欄位辨識全部為規則式手寫邏輯（正規表示式、字元組成比例、次數統計/眾數、集合去重計數、模運算週期分桶、固定閾值決策），不是機器學習模型。完整規則表見 `README.md`。
- **架構決策：FastAPI + 獨立微服務**（2026-07-15 確認，已寫入 README「計畫架構」章節）：理由是無狀態單一職責的 I/O bound 服務適合原生 async、Pydantic 驗證與自動 OpenAPI 文件對未來被平台其他服務呼叫有直接幫助、不需要 Django 的 ORM/Admin/Session。服務邊界維持獨立，透過 HTTP API 與平台其他子專案串接，不預期併入同一 codebase。**「獨立服務」是技術/部署層面決定，不等於「獨立維護」（組織/人力層面）**——目前這兩者剛好是同一人，但概念上是分開的。獨立服務架構的技術義務（API 版本相容性、範本本機檔案儲存在多實例/獨立部署時需改共用儲存）目前成本為零（尚未真正部署、無其他服務依賴），但 API 形狀（template JSON schema、端點路徑）之後不宜隨意變更。
- **`/api/preview` 回傳完整 `employees` 陣列**（非僅樣本），供前端單日預覽自由切換任一天，不需額外呼叫後端。
- **確認 Modal 區塊列語意編輯為「完全通用版」**：每一列可自由選 5 種類型（shift/shift_start/shift_end/shift_string/metadata），直接對應 `backend/parser.py` 支援的所有真實排版（非簡化版，因為 6 份真實檔案中有 2 份就是「簡化版」原本要排除的特殊排版）。
- **前端無框架**：原生 HTML/CSS/JS，FastAPI 直接掛載靜態檔案，未使用 React/Vue 等前端框架，亦無 build step。
- **員編判定門檻**：命中率 ≥ 50% 的區塊數才採用該欄位，寫死於 `backend/heuristics.py`，尚未依真實資料調整。
- **範本儲存結構**：`templates/{scope_id}/{template_id}.json`，`scope_id` 由使用者上傳頁手動輸入（無帳號系統，路徑安全字元已驗證防路徑穿越）。
- **姓名長度限制**：判定為姓名候選的字串最長 20 字元。
- **Gantt 圖表原生需求**：下載輸出檔案為 openpyxl/ODF 原生圖表物件；網頁預覽為 CSS 繪製的長條圖（非圖表庫），僅供瀏覽/列印使用，不影響下載檔案的原生圖表需求。
- **ODS 輸出固定以 `白霧.ods` 作為圖表骨架來源**（`generator.generate_ods` 既有行為），資料本身來自使用者上傳檔案解析結果，只有 Gantt 圖表容器沿用白霧範本。
- **檔案安全**：解析時只當純資料讀取，不執行巨集、不追蹤外部連結；上傳檔案有 20MB 大小上限與副檔名白名單。
- **Git 遠端倉庫**：`https://github.com/jamessun0919-ops/Shift-schedule`（已設為 origin）。
- **推送規則**：僅推送紀錄檔案（交接文檔/工作日誌/對話紀錄/README）與原始碼（`backend/`、`frontend/`、`prototype/`、`tests/*.py`、`requirements.txt`）；測試/範例用資料檔案（`.ods`/`.xlsx`）與 `__pycache__`、`templates/`（測試期間產生的範本資料）依 `.gitignore` 不推送。
- **開發環境注意事項**：Windows 終端機 codepage（cp950/Big5）會讓中文參數/輸出在 bash 顯示亂碼，這是純顯示層問題，非資料損毀——驗證資料正確性請用 Python 檔案 I/O 或直接讀檔案內容比對，不要僅憑終端機顯示判斷。
