## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel 檔案，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估）。**輸出格式決策：僅提供 Excel（.xlsx）輸出，不提供 ODS 輸出**；上傳輸入端仍支援 .xlsx / .ods。**部署決策：單一 Render Web Service 承載前端＋後端，暫不使用 Docker、暫不使用付費持久化磁碟**（詳見下方）。

### 部署上線：Render
- **正式網址**：`https://shift-schedule-b5ne.onrender.com`（Render Free tier）。健康檢查：`GET /api/health` → `{"status":"ok"}`。使用手冊：`/manual.html`。
- **架構決策**：單一 Render Web Service 同時承載前端靜態檔＋FastAPI API（不拆分 Vercel，因 Vercel serverless 無持久化磁碟、與本專案「範本存本機檔案系統」機制不相容）。採 Render 原生 Python 路線，**暫不使用 Docker**（依賴極輕，Docker 價值在未來搬遷可攜性，非現階段必要）。
- **`render.yaml`**（repo 根目錄）：
  ```yaml
  services:
    - type: web
      name: shift-schedule
      env: python
      plan: free
      buildCommand: pip install -r requirements.txt
      startCommand: uvicorn backend.main:app --host 0.0.0.0 --port $PORT --proxy-headers
      autoDeploy: true
      envVars:
        - key: PYTHON_VERSION
          value: 3.11.9
  ```
  `PYTHON_VERSION` 釘選穩定版本（本機開發環境是過新的 3.14.5，不確定 Render 平台是否已支援，程式碼未用 3.12+ 語法故選保守版本）。`--proxy-headers` 是本輪 rate limiting 工作新增的關鍵旗標（見下方）。
- **兩項已知限制，皆已決策定案**：(1) **範本持久化**——demo 階段接受 Free tier 無 Persistent Disk 導致 `templates/` 於重新部署/閒置回收後清空，待商轉時升級付費方案＋掛 Persistent Disk 解決；(2) **存取控制**——demo 階段暫不處理 `scope_id` 無歸屬驗證問題，待日後接入餐飲平台主體時，`scope_id` 改由平台的帳號密碼機制產生，屆時再實作。**這兩項是產品決策已定案，非技術阻塞**。

### 系統架構檢視與三項修正（本輪核心工作）
- **背景**：Render 上線、正式對外公開後，通讀 `backend/main.py` 全部端點進行架構檢視，發現四項問題，逐一討論後修正三項：

1. **API rate limiting**（已修正）：`/api/analyze`／`/api/preview`／`/api/convert` 三個實際執行解析/產生檔案的端點，用 `slowapi` 加上 **20 次/分鐘（依 client IP）** 限制；`/api/templates` CRUD 與 `/api/health` 不限流。**門檻設計原則**：作為「濫用斷路器」而非一般使用配額——實測正常使用（新分析流程約 4~6 次呼叫、已存範本流程約 3 次）遠低於 20 次，只會擋到明顯異常的重複呼叫。**關鍵技術細節**：Render 邊緣代理若不處理，`slowapi` 預設的 `get_remote_address` 會抓到 Render 代理的 IP 而非真實使用者 IP，導致「所有使用者共用同一組額度」（比使用者原本擔心的「同店共用 IP」更嚴重）。已在 `render.yaml` 的 startCommand 加上 `--proxy-headers` 修正。**⚠️ 需使用者手動同步**：因為 Render 服務當初是手動貼上 Start Command（非透過 `render.yaml` Blueprint 自動套用），需到 Render dashboard 把 Start Command 改成 `uvicorn backend.main:app --host 0.0.0.0 --port $PORT --proxy-headers`，否則限流會誤判所有使用者為同一來源。
2. **例外訊息外洩**（已修正）：`backend/main.py` 的 `/api/analyze`／`/api/preview`／`/api/convert` 原本 `except Exception as e: raise HTTPException(422, f"...：{e}")` 直接把 Python 例外內容回傳前端。**查證非 debug 殘留**：git log 確認此設計自 2026-07-15 建立後端的第一個 commit 就存在。修正為回傳篩選過的固定提示文字，真實例外內容改用 `logging.exception()` 寫伺服器端 log（`logging.basicConfig` 已加在 `backend/main.py` 頂部，Render 會自動收集 stdout/stderr）。
3. **同步阻塞運算佔用 async 事件迴圈**（已修正）：三個端點原宣告 `async def` 但內部完全沒有 `await`（`openpyxl` 解析/產生圖表是同步運算），等於白白讓阻塞運算佔住 FastAPI 唯一的事件迴圈、拖累其他所有請求。改為宣告一般 `def`，FastAPI/Starlette 自動丟進執行緒池執行，函式內部邏輯不變。
4. **無伺服器端 log／可觀測性**（部分改善，非本輪目標）：上述第 2 項修正的副作用讓三處例外路徑現在有 log，但**不是**全面的請求層級 log（成功請求、耗時、呼叫頻率皆無記錄）。若之後「使用者回報異常但無法重現」且不是走這三個例外路徑（例如解析「成功」但結果不正確），現有 log 仍幫不上忙。

- **驗證方式**：全程用 FastAPI `TestClient`（非瀏覽器）——rate limit 測試（連續呼叫 21 次確認第 21 次起 429，`/api/health` 連續 30 次不受影響）、例外訊息測試（觸發三處例外路徑確認回應不含原始例外內容）、三端點成功路徑端對端測試（真實範例檔案：分析→試解析→下載，確認回傳 xlsx 內容），以及 `tests/verify_engine.py` 六份範例迴歸測試，全數通過無回歸。
