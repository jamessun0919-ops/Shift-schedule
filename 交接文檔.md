## 專案目標 (Project Goal)

做一個泛用性網頁工具，把不同公司/品牌的格狀月排班表 Excel/ODS（欄位配置不固定）轉換成每日一分頁、橫向長條圖顯示的排班表 Excel/ODS，解決格狀班表不利於每日/即時人力控管的問題。**本工具是更大餐飲管理平台的子模組**，需具備多品牌/多店家泛用性，但不自建帳號/身分驗證系統。**架構決策：維持獨立微服務（FastAPI），與其他子專案透過 HTTP API 串接，不預期併入其他子專案的 codebase**（待平台主體技術棧確定後再視需求重新評估）。

## 目前的瓶頸或停頓點 (Current Blocker/Status)

**唯一的實質阻塞點**：ods 下載檔案甘特圖的姓名軸／時間軸排版需求（同 xlsx 的兩項新需求：姓名由上到下、時間軸移到上方）尚未處理。

- ODS 的甘特圖是套用 `白霧.ods` 範本裡既有的 LibreOffice 手工圖表物件（`generate_ods()` 只改寫既有圖表 series 的儲存格範圍與座標軸 min/max，從不重建圖表結構），跟 xlsx 用 openpyxl 從零程式化組出圖表完全是兩套機制，xlsx 那邊查到的兩個 bug 不代表 ods 也有同樣問題。
- 查證 ODF 圖表 XML 發現：姓名軸（x 維度）已有一個**非預設值**的 `chart:reverse-direction="true"`，可能代表原作者當初手動建立範本時已經處理過姓名順序（也可能不是、原因未知）；時間軸（y 維度）要移到上方需要調整 `chart:axis-position`，但這個屬性在 ODF 1.2 spec 裡對「值軸相對類別軸的定位」語意，沒有把握單憑記憶推導到 100% 正確。
- **本環境沒有 LibreOffice 可以實際渲染/開啟 ODF 檔案驗證外觀**，直接照記憶去改這類圖表 XML 屬性有把圖表改壞（甚至無法開啟）的風險，且改錯了也無法自行發現。
- 已請使用者先下載目前的 ods 檔案，用 LibreOffice/Excel 實際開啟看姓名順序與時間軸位置目前長什麼樣子，回報後再決定：(a) 完全不用改（如果 `reverse-direction=true` 其實已經是正確方向）、(b) 只需要調整時間軸位置、(c) 兩者都要調整。

其餘皆為非阻斷性已知限制：
1. 員編欄位偵測門檻（≥50%填寫率）僅在有限真實案例上驗證過，門檻本身是否合理尚待更多真實資料佐證。
2. N 段班（3 段班）僅完成程式邏輯驗證（合成資料），真實 3 段班班表的啟發式偵測準確度尚未驗證。
3. 部署方式尚未決定，且**尚未展開討論**（使用者要求先完成本地測試與 UI 修改，部署相關明確延後）。

## 關鍵設定與上下文 (Key Context & Rules)

- **技術定位**：欄位辨識全部為規則式手寫邏輯，不是機器學習模型。完整規則表見 `README.md`。
- **架構決策：FastAPI + 獨立微服務**（2026-07-15 確認，已寫入 README）。
- **範本 JSON schema 新增 `display` 區塊（本輪）**：`{"axis_start": int, "axis_end": int}`（整點小時，0~24）。四個讀取點都要 fallback 8/24（舊範本無此欄位時）：`frontend/app.js` 的 `renderDayGantt()`、`backend/generator.py` 的 `generate_xlsx()`/`generate_ods()`、`backend/heuristics.py` 的 `guess_axis_range()` 找不到有效資料時。**未來若要調整這個 schema，四個讀取點要一起改，容易漏改其中一處。**
- **openpyxl BarChart 的軸對應關係（本輪踩到的坑，務必記住）**：`chart.x_axis` 不論 `chart.type` 是 `"bar"`（橫向）或 `"col"`（縱向），永遠序列化成 `<c:catAx>`（類別軸）；`chart.y_axis` 永遠是 `<c:valAx>`（數值軸）。這跟畫面上長條實際橫向或縱向排列**無關**——`chart.type` 只控制 `<c:barDir>`，不影響 x_axis/y_axis 分別對應哪個角色。之前的 bug 就是把數值/時間相關設定誤放在 `x_axis` 上。未來若要再調整這個圖表，時間/數值相關設定一律設在 `y_axis`，姓名/分類相關設定一律設在 `x_axis`。
- **openpyxl `chart.set_categories()` 呼叫時機**：必須在所有 `chart.add_data()` 呼叫**之後**才呼叫，因為它是遍歷 `chart.series` 逐一設定 `.cat`，呼叫當下如果 series 是空的就是無效呼叫，不會報錯但分類資料永遠不會出現在輸出檔案裡。
- **ODS 圖表座標軸相關 XML 結構（本輪查證，供下次接手 ods 排版需求參考）**：`白霧.ods` 每個 `Object N/content.xml` 內，`<chart:axis chart:dimension="x">` 是姓名/類別軸（引用某個 `style:style`，該 style 目前有 `chart:reverse-direction="true"`），`<chart:axis chart:dimension="y">` 是時間/數值軸（引用另一個 style，含 `chart:minimum`/`chart:maximum`，本輪的座標軸範圍功能已會動態改寫這兩個值）。姓名軸與時間軸的排版（順序/位置）尚待查證 ODF spec 精確語意後才能安全修改，見上方「目前的瓶頸」。
- **`/api/preview` 回傳完整 `employees` 陣列**，供前端單日預覽自由切換任一天。
- **確認 Modal 區塊列語意編輯為「完全通用版」**：每一列可自由選 5 種類型（shift/shift_start/shift_end/shift_string/metadata）。
