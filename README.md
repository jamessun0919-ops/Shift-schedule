# 班表格式轉換工具

餐飲業月排班表格轉換為每日橫向長條圖排班表的網頁工具。

## 專案目標

輸入不同公司/品牌的格狀排班 Excel/ODS（員工 × 日期，欄位配置不固定），輸出橫向長條圖排班 Excel/ODS（每日一分頁，含 Excel 原生 Gantt 式長條圖），解決格狀班表不利於每日/即時人力控管的問題。

## 計畫架構

- **輸入**：Excel/ODS 檔案，含外場/內場格狀原始輸入表。
- **欄位辨識**：啟發式猜測（結構特徵統計+使用者確認畫面），處理主要特徵（姓名/日期/員工編號/班別）欄位不固定的問題。次要特徵（行銷/活動附註）不做自動辨識，改於輸出的每日頁面（網頁預覽與檔案分頁）保留空白欄位供人工填寫。
- **範本儲存**：JSON 儲存層存欄位對照範本（`templates/{scope_id}/{template_id}.json`），使用者上傳時選擇範本以重用設定。
- **後端**：Python（FastAPI + openpyxl），無狀態 REST API（上傳/分析兩次無留存檔案，不用 session），解析每人每日上下班時間，產生每日分頁 + 原生長條圖。
- **前端**：單頁式上傳頁面 + 欄位對照確認 Modal（僅在使用者按下「確認」時才呼叫一次後端試解析，Modal 內微調不即時發請求）+ 下載頁面。
- **輸出**：轉換後的 Excel/ODS 檔案（每日一分頁，包含原生 Gantt 堆疊長條圖）。

### 架構決策：後端框架與服務邊界（2026-07-15）

- **框架選擇**：FastAPI（而非 Django/Flask）。理由：本工具是無狀態、單一職責的檔案轉換服務，I/O bound（讀檔/解析/寫檔）適合原生 async；Pydantic 驗證與自動 OpenAPI 文件（`/docs`）對「未來被平台其他服務呼叫」的整合成本有直接幫助；且不需要 Django 的 ORM/Admin/Session（目前無資料庫、無帳號系統）。
- **服務邊界**：本專案**維持獨立微服務**，未來與其他子專案（更大的餐飲資訊平台）整合時，透過 HTTP API 串接，不預期將程式碼併入其他子專案的 codebase。
- **待確認/待改寫的觸發條件**：此決策待平台主體技術棧與整合方式確定後重新評估。若平台整合方向改為「併入同一個 codebase」而非「API 串接」，屆時需重新評估框架選擇（現有 FastAPI 邏輯需要整包搬遷/重寫，非本次決策範圍）。
- **已知的獨立服務技術義務（尚未發生，但架構上遲早要面對）**：範本目前存於本機檔案系統（`templates/`），若未來真的獨立部署且需要多實例/水平擴展，本機檔案非共用狀態，需改為共用儲存或資料庫；API 一旦被其他模組依賴，欄位對照 JSON schema 與端點路徑需注意向後相容，不隨意變更。

## 已完成進度

- **啟發式欄位辨識模組 (`backend/heuristics.py`)**：日期列嗅探（唯一值+數量門檻）、姓名欄位偵測、員工區塊列數推估、區塊內姓名列位置偵測、員工編號欄位偵測。全部改為規則式統計判斷（字元組成比例、重複次數、眾數、週期分桶），不使用寫死的職稱關鍵字清單，也不使用機器學習模型——詳見文末〈欄位辨識規則技術參考〉。
- **通用解析模組 (`backend/parser.py`)**：支援任意班別數的區塊列語意（N 段班，非寫死 2 班）、單行文字列（自動解析含休息時間的兩段式排班）、跨列起訖堆疊格式。區塊邊界改用「錨點+週期分桶」方式偵測，不受表頭雜訊列（如星期標題列）干擾。補足 ODS 日期格式儲存格（`vtype == "date"`）的解析。
- **通用產生器模組 (`backend/generator.py`)**：`_get_n_shifts` 動態估計範本最大 Shift 段數，防止圖表產出時報越界錯誤。
- **回歸測試與驗證**：`tests/verify_engine.py` 涵蓋既有基準檔（`白霧.ods`）與 5 份不同格式真實班表（範例班表 1～5），驗證啟發式偵測、解析、輸出 XLSX Gantt 圖轉換全數正確（姓名、班次時間、可用時之員編皆與原始資料逐一核對）。另以合成資料驗證 3 段班（N-shift）情境下程式邏輯正確。
- **FastAPI 後端伺服器 (`backend/main.py`)**：無狀態 REST API，含 `/api/analyze`（上傳分析）、`/api/preview`（試解析健康檢查）、`/api/convert`（產生下載檔，xlsx/ods）、範本 CRUD（`/api/templates/...`）。已用真實檔案（範例班表1.ods）跑通 end-to-end：分析→預覽→轉換（xlsx 與 ods 皆驗證）→範本建立/查詢/更新/刪除全數正確；邊界防呆（不支援副檔名、空檔案、壞檔案、壞 JSON、超過 20MB、路徑穿越攻擊）皆回傳正確錯誤碼，不會使伺服器崩潰。
- **單頁面前端網頁 (`frontend/index.html` / `style.css` / `app.js`)**：原生 HTML/CSS/JS（無框架），由 FastAPI 掛載為靜態網站（同一個 process、同一個 port）。含上傳表單、範本選擇（讀取已存範本）、欄位對照確認 Modal（原始資料格線預覽 + 通用區塊列語意編輯器，支援 5 種列類型自由切換：班別/僅上班/僅下班/文字班別/員編）、另存為範本、單日排班橫向長條圖預覽（1-31 日切換）、`window.print()` 列印（獨立列印樣式，隱藏非必要區塊）、轉換下載（xlsx/ods）。已用 Playwright 實際跑過瀏覽器 end-to-end（上傳→分析→修改欄位對照→試解析→切換預覽日期→另存範本→重新整理後範本仍在→列印預覽→轉換下載，下載檔案已驗證可用 openpyxl 開啟且內容正確），過程中發現並修復一個 CSS Grid overflow bug（確認 Modal 右欄位對照表單因內部寬表格撐開而位移到畫面外）。
- **確認欄位對照 Modal 的 UI 潤飾（2026-07-16，第一輪）**：原始資料預覽格線改用試算表座標（欄 `A,B,C`／列 `1,2,3`）；三個定位欄位（日期所在列、姓名所在欄、上下班起始格）新增「從表格點選」自動帶入，滑鼠懸停即時預覽高亮；欄位對照文字改為口語化提問。皆為前端顯示層調整，範本 JSON schema 不變。
- **工作日誌與對話紀錄**：每日彙整於 `工作日誌.md`／`對話紀錄.md`，交接文檔同步更新。

## 欄位辨識規則技術參考

> 本專案的「欄位辨識」全部為**規則式（rule-based）手寫邏輯**，不使用機器學習模型（無訓練資料、無模型權重、無梯度下降）。設計理念與 Mailchimp／Stripe／銀行對帳單匯入工具的「欄位嗅探＋人工確認」模式相同：啟發式先猜，猜不準或沒把握時，交給使用者在確認畫面手動修正，不追求全自動辨識。這個章節記錄實際用到的技術手法與各欄位的判斷規則，供後續查詢或作為技術文件參考。

### 用到的技術手法

| 技術 | 用途 | 說明 |
|---|---|---|
| 正規表示式（Regex） | 結構性樣式比對 | 中文姓名 2-4 字、日期格式（`7月1日`、`1/1`）、時間格式（`HHMM-HHMM`） |
| 字元組成比例統計 | 區分姓名／員編 | 計算字串中數字佔全部英數字元的比例（`digit_ratio`），數字佔比高判定為員編、低判定為姓名 |
| 次數統計＋眾數選擇 | 決定欄位、決定區塊列數 | 用 `collections.Counter` 對候選欄位/候選間距計數，取出現次數最多者（不是取平均、不是加權，就是簡單次數統計） |
| 集合去重計數 | 區分姓名／類別標籤（職稱） | 職稱（如「服務員」）在不同員工間會重複出現，姓名幾乎不重複；比較各候選位置「相異值數量」，取最多者為姓名欄位 |
| 模運算週期分桶 | 對齊員工區塊邊界 | 以第一個結構上像姓名的儲存格為錨點，其餘列依 `(row - 錨點) % 區塊列數` 分桶，不受表頭雜訊列（如星期標題列）造成的位移影響 |
| 固定閾值決策 | 是否採信猜測結果 | 例如「候選欄位命中率 ≥ 50% 才採用，否則不猜」，門檻寫死於程式，非學習所得參數 |
| 合理性驗證（非統計推論） | 解析後的健檢 | 檢查時間是否落在 0–24 區間、結束時間是否不早於起始時間，超過 30% 異常則標記整份解析不健康 |

### 各欄位辨識規則

| 欄位/項目 | 判斷依據 | 具體規則 | 對應函式 |
|---|---|---|---|
| 日期標頭列 | 該列符合日期/天數格式的儲存格數量與相異值數量 | 符合格式數 ≥ 15 且相異值 ≥ 10 的列中，取符合數最多者；找不到則退回第 4 列 | `guess_template` |
| 每天欄位數（cols_per_day） | 日期欄位間的欄距 | 取前幾個日期欄位間距的眾數 | `guess_template` |
| 姓名欄位 | 各候選欄位（第1~5欄）中結構性姓名候選數量 | 中文 2-4 字，或英數字元且數字佔比 < 50%；取候選數最多的欄位 | `is_name_like` / `find_name_rows` |
| 姓名列（區塊內偏移） | 各候選列位置的相異姓名值數量 | 以錨點分桶後，取相異值數量最多的桶為姓名列位置（職稱等類別標籤會因重複出現而相異值較少，即使該職稱全店僅一人擔任也一樣會被比較掉，因為同一桶內其他職稱仍會重複） | `find_block_anchors` |
| 員工區塊列數（expected_rows） | 姓名候選列之間的間距 | 取候選列間距的眾數；超過 5 視為異常，退回單列解析 | `guess_template` |
| 員工編號欄位 | 區塊中「行政/總計列」（區塊最後一列）各欄位符合員編特徵的比例 | 不含中文、數字佔比 ≥ 50%、長度 ≥ 3；命中率 ≥ 50% 的區塊數才採用該欄位，否則不猜測（欄位留空） | `is_id_like` / `guess_template` |
| 班別列語意（N 段班） | 區塊列數扣除最後一列（行政/總計列）後的其餘列數 | 最後一列固定視為行政/員編列，其餘列依序編號為班別 0、1、2…N，不寫死班別數量 | `guess_template` |
| 表頭/統計字樣排除 | 固定詞彙表比對 | 「姓名」「日期」「星期」「總計」「說明」等結構性表頭詞彙（跨品牌通用，非職稱），命中即排除，不計入姓名或員編候選 | `is_header_keyword` |
| 健康檢查 | 解析後全體班次時間的異常比例 | 時間需落在 0–24 內、結束時間不早於起始時間；異常比例 > 30% 判定整份解析不健康 | `perform_health_check` |

**刻意不採用關鍵字清單的地方**：職稱辨識（店長／領班／服務員…）原本用寫死的中文職稱清單排除，已改為「重複次數 + 相異值統計」的結構性判斷（見上表「姓名列」）。原因：職稱清單只涵蓋餐飲業，換到其他產業或品牌就會失效；統計特徵不需要為每個新品牌維護一份新清單。
